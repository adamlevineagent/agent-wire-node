# Question Pyramid Architecture

## Core Insight

A Knowledge Pyramid is not a data compression tree. It is a sequence of questions asked about a corpus, where each layer's questions are asked about the answers from the layer below.

The "pyramid" that users navigate is a materialized view over a question-answer graph. Different question sequences produce different pyramids from the same base data. The tree structure is a rendering choice, not a data structure.

## Foundational Principles

### 1. L0 is the only extraction layer

L0 nodes are the atoms. Each source document (file, page, transcript) is extracted independently, in a vacuum, by an LLM. The extraction produces topics, entities, decisions, corrections — whatever the document contains. L0 extraction is deterministic given the same document and prompt. L0 nodes are permanent and immutable (until the source document changes).

### 2. Everything above L0 is a question

Each node above L0 exists because a question was asked. The question might be:
- "What are the distinct concepts across all L0 topics?" → produces concept clusters
- "What resources are shared between these concepts?" → produces web edges
- "What changed over time across these documents?" → produces temporal evolution
- "What alternatives were considered and rejected?" → produces rejection analysis
- "What are the cross-cutting failure modes?" → produces resilience analysis

The answer to each question is stored as edges in a graph, not as a fixed tree.

### 3. Questions compose vertically

L1 questions are asked of L0 atoms. L2 questions are asked of L1 answers. L3 questions are asked of L2 answers. The depth of the pyramid equals the depth of the question chain.

A shallow pyramid (L0 → L1 → apex) means only first-order questions were asked. A deep pyramid means questions were asked about questions about questions — each layer adding genuine understanding.

### 4. Questions compose horizontally

Multiple independent questions can be asked at the same depth. "What are the concepts?" and "What is the timeline?" are both L1 questions. They produce independent overlays. Either can exist without the other.

A wide pyramid means many independent questions were asked at L1. A narrow pyramid means few questions, but deeper chains.

### 5. Each question's answer can feed later questions

The output of one question becomes available input for subsequent questions. The concept clusters make the webbing question smarter. The webbing edges make the temporal question smarter. Questions can declare dependencies on prior answers:

```
Q: "What are the active design tensions?"
Requires: concept_edges, temporal_edges, rejection_edges
```

### 6. The tree view is a materialization

Users navigate a tree because trees are intuitive. But the underlying data is a graph of edges:
- L0 topic → concept (many-to-many)
- concept → web edge → concept
- concept → temporal edge → concept
- L0 topic → temporal position
- concept → domain (L2 grouping)

The tree is generated by picking a root question (the apex question), tracing its dependencies, and rendering the resulting DAG as a navigable hierarchy. Different root questions produce different trees.

## Architecture

### Edge Types

All knowledge above L0 is stored as typed edges:

| Edge Type | Connects | Example |
|-----------|----------|---------|
| `concept_member` | L0 topic → concept | "D-L0-003 topic 2 belongs to concept 'Credit Economy'" |
| `web_edge` | concept ↔ concept | "Credit Economy shares `credit_transactions` table with Auth System" |
| `temporal_edge` | L0 → L0 | "D-L0-006 supersedes D-L0-001 on OCU model" |
| `rejection_edge` | concept → rejected alternative | "Action-based OCU was rejected in favor of consumption-value OCU" |
| `domain_member` | concept → domain | "'Credit Economy' belongs to domain 'Governance & Economics'" |
| `question_answer` | question → materialized node | "Q: 'What are the concepts?' → answer node L1-003" |

### Question Definition

A question is defined by:
- **ask**: Natural language description of what to find out
- **instruction**: LLM prompt that operationalizes the question
- **input**: What data feeds this question (L0 atoms, or answers from prior questions)
- **produces**: What edge type this question creates
- **uses** (optional): Which prior question answers to include as context

### Materialization

When a user opens a pyramid, the system:
1. Starts from the apex question
2. Traces the dependency chain of questions
3. For each question, gathers the relevant edges
4. Renders the edges as a navigable tree where:
   - Each node is a materialized answer
   - Children are the inputs that fed the answer
   - Siblings are parallel answers at the same depth
   - Web edges appear as cross-links between siblings

### L0 Many-to-Many

A single L0 document can contribute to multiple L1 concepts. A foundational design doc might connect to 12 of 15 concepts. A narrow audit report might connect to 2. The children array on L1 nodes contains ALL L0s that contribute, not just the ones a clustering algorithm assigned.

This means:
- Drilling into "Credit Economy" shows every document that discusses credits, not just the ones clustered there
- A central document like "platform-strategy.md" appears as a child of many L1 nodes
- The UI shows the full web of paths to every topic, not just the single path the clustering picked

## Migration Path

### Current State (v2.0)
- L0 extraction → thread clustering → thread synthesis → recursive clustering → apex
- Tree structure with single-parent children
- Web edges stored separately in `pyramid_web_edges`
- Fixed question sequence hardcoded in YAML

### Transition (v3.0)
- Keep current pipeline as the default question sequence
- Add horizontal coordination pass between L0 and L1 (concept discovery)
- Allow L0s to appear in multiple L1 children arrays (DAG)
- Store question metadata alongside materialized nodes
- Web edges, temporal edges, rejection edges become first-class overlay types

### Target State (v4.0)
- Questions defined declaratively in YAML
- Question dependencies resolved automatically
- Multiple question sequences produce multiple views from same L0 base
- Materialization is on-demand: "show me the security pyramid" generates a view
- Agents can add new questions: "what are the performance bottlenecks?" adds an overlay without rebuilding

## Question Sequence Examples

### Default Knowledge Pyramid
```
L1: "What are the 8-15 distinct concepts across all L0 topics?"
L1: "What resources are shared between sibling concepts?" (webbing)
L2: "What are the 3-5 architectural domains that group these concepts?"
L2: "What connects these domains?" (webbing)
Apex: "Synthesize all domains into a unified understanding"
```

### Security Audit Pyramid
```
L1: "What are the auth/security mechanisms across all L0s?"
L1: "What are the trust boundaries and token flows?"
L2: "What are the attack surfaces?"
L2: "What mitigations exist for each attack surface?"
Apex: "What is the overall security posture and what's missing?"
```

### Temporal Evolution Pyramid
```
L1: "What are the major decisions across all L0s, ordered by date?"
L1: "Which decisions supersede earlier ones?"
L2: "What are the active design tensions (unresolved contradictions)?"
Apex: "What is the current state of the system's design and what's still evolving?"
```

### Onboarding Pyramid
```
L1: "What are the major subsystems a new developer needs to know?"
L1: "What are the 'gotchas' — things that surprise people?"
L2: "What's the recommended reading order?"
Apex: "One-page developer orientation"
```

## Implications

### For the Build Pipeline
The YAML chain definition evolves from a sequence of primitives (extract, classify, synthesize) to a sequence of questions. Each question is a step. The executor runs questions in dependency order, storing edges.

### For the Stale Engine
When a source document changes, L0 re-extraction triggers. But instead of rebuilding the whole tree, the system re-runs only the questions whose inputs changed. If a document about auth changes, only questions that touch auth-related L0 topics are re-asked.

### For Agent Contributions
An agent working on the codebase can contribute a new question: "What are the undocumented assumptions in the auth system?" This adds an overlay without modifying the base pyramid. Other agents benefit from the answer without needing to know who asked the question.

### For the FAQ Engine
FAQs are just questions that users have already asked and agents have already answered. Each FAQ entry is a question with a materialized answer, connected to the L0 topics that informed it. The FAQ engine IS the question pyramid — just with user-submitted questions instead of system-defined ones.

### For Multi-User
One L0 base, many question sequences. Your team sees the default pyramid plus a security overlay. An external auditor sees the default pyramid plus a compliance overlay. An investor sees a business model pyramid. Same atoms, different questions, different views.
