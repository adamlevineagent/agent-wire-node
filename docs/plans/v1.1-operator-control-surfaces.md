# Wire Node v1.1 ‚Äî Operator Control Surfaces

## Context

The v1 UX ship delivered observation across 9 tabs. But the app is fundamentally **read-only for fleet operations**. The Wire server has a rich operator control plane (`/api/v1/operator/*`) that the desktop app doesn't surface. A human directing a swarm needs to **control**, not just watch.

---

## Phase 0: Wire Server Auth Fix (HARD GATE ‚Äî blocks Phases 1, 3, 5c)

**Problem:** Most operator management endpoints (`/operator/agents/[id]/status`, `/controls`, `/token/regenerate`, `/contributions`, `/queries`, `/overview`, `/review-queue`, `/review-feed`) use `requireOperatorApi()` which reads from an `op-session` cookie. The desktop app stores the same token in `auth.operator_session_token` but sends it as `Authorization: Bearer`. Cookie-based auth from Tauri is fragile (SameSite, domain scoping).

**Solution ‚Äî Wire server change (~15 lines):** Add a Bearer fallback path to `requireOperatorApi()` in `GoodNewsEveryone/src/lib/server/operator-auth.ts`. **Do NOT copy the `requireOperatorOrAgentAuth()` pattern** ‚Äî that function takes a `Request` parameter, but `requireOperatorApi()` takes zero parameters and uses `next/headers` imports.

Implementation:
1. Import `headers` from `next/headers` (same module `cookies` comes from)
2. When no `op-session` cookie is found, read `(await headers()).get('authorization')`
3. If it starts with `Bearer `, extract the token
4. **Token discrimination:** if token starts with `gne_live_`, return 401 immediately (agent tokens cannot authenticate as operators)
5. Call the existing `verifyOperatorSession(token)` from `wire-auth.ts` (does hash + lookup in `wire_operator_sessions` + legacy `wire_operators.session_token_hash`)
6. Return `{ operatorId, email }` on success, fall through to 401 on failure
7. **Legacy JWT path (path 3) is NOT needed** in the Bearer fallback ‚Äî desktop clients always get opaque session tokens from `/operator/auth/session`, never JWTs

This makes all `/operator/*` endpoints callable via the existing `operatorApiCall` in the desktop app. **No new Tauri command needed.** Both `wireApiCall` and `operatorApiCall` are already available in AppContext ‚Äî no new plumbing required for frontend.

**Files:**
- `GoodNewsEveryone/src/lib/server/operator-auth.ts` ‚Äî add Bearer fallback to `requireOperatorApi()`

**Smoke test:** After deploying, call `operatorApiCall('GET', '/api/v1/operator/overview')` from the desktop app and confirm 200 response.

**NO downstream phase starts until this is deployed and verified.**

## Tab-by-Tab Control Additions

### 1. Fleet > Fleet Overview ‚Äî Agent Management

**Currently:** Read-only agent roster with online status.

**Add:**
**Dual-auth pattern:** FleetOverview currently destructures only `wireApiCall` from context (for roster + pulse). Phase 1 must also destructure `operatorApiCall` for agent detail, controls, and status endpoints. Both are already available in AppContext ‚Äî no new plumbing needed.

- **Agent detail drawer** ‚Äî click agent card to open SlideOverPanel with: profile (from `GET /operator/agents/[pseudo_id]`), contribution history (`GET /operator/agents/[pseudo_id]/contributions`), query history (`GET /operator/agents/[pseudo_id]/queries`), reputation breakdown, controls. All use `operatorApiCall` (after Phase 0 Bearer fallback). The detail response includes the internal UUID at `detail.agent.id` (nested under the `agent` object, NOT a top-level `agent_id` field) ‚Äî cache it for archive/unarchive operations.
- **Pause/Resume agent** ‚Äî toggle in drawer. Calls `operatorApiCall('PATCH', '/api/v1/operator/agents/[pseudo_id]/status', { status: 'active' | 'paused' })`.
- **Archive agent** ‚Äî button with **impact preview** (Pillar 23): show assigned task count, active intent count, pending contributions. Then confirm. Calls `operatorApiCall('POST', '/api/v1/wire/agents/archive', { agent_id: "<UUID>" })` ‚Äî note: `agent_id` is the internal UUID from the detail response, NOT the pseudo_id.
- **Unarchive agent** ‚Äî for archived agents. `operatorApiCall('DELETE', '/api/v1/wire/agents/archive', { agent_id: "<UUID>" })`.
- **Revoke agent** ‚Äî destructive, permanent. **Full impact preview** (Pillar 23): show active contributions count, pending royalties, downstream citation count, assigned tasks, last-seen timestamp. Double confirm. Calls `operatorApiCall('PATCH', '/api/v1/operator/agents/[pseudo_id]/status', { status: 'revoked' })`.
- **Regenerate token** ‚Äî button with **impact preview** (Pillar 23): show last-seen timestamp, active connections hint. Shows new token ONCE after generation. Calls `operatorApiCall('POST', '/api/v1/operator/agents/[pseudo_id]/token/regenerate')`.
- **Agent controls panel** ‚Äî in drawer: toggle `can_contribute`, `can_query`, `can_rate`, `can_purchase`, set `daily_query_budget`, set `spend_cap_pct`, toggle `contribution_hold`, set `domain_restrictions[]`. Calls `operatorApiCall('PATCH', '/api/v1/operator/agents/[pseudo_id]/controls', { ...fields })`.
- **Create new agent** ‚Äî button at top. Form: name (required). On submit, calls `wireApiCall('POST', '/api/v1/register', { name, operator_email: state.email })` (register is **unauthenticated** ‚Äî either auth path works, no Phase 0 dependency). Returns `api_token` and `agent_id` (UUID) ‚Äî show token once in a modal ("Save this token now, it won't be shown again"), cache agent_id for archive operations. Handles 409 (name taken).
- **Filter/search agents** ‚Äî client-side filter by status (active/paused/archived), online status, name text search.
- **Bulk archive** ‚Äî checkbox per card, "Archive selected" button. Loops `POST /wire/agents/archive` per agent (no bulk endpoint exists).

### 2. Fleet > Tasks ‚Äî Full Task Management

**Currently:** Kanban with broken Claim button (sends `{ status: "claimed" }` instead of `{ action: "claim" }`).

**Fix existing bugs first:**
1. Change claim body from `{ status: 'claimed' }` to `{ action: 'claim' }` (the HTTP method PATCH works fine ‚Äî server aliases PUT as PATCH)
2. Rename `description` to `context` in `WireTask` interface and rendering (server returns `context`, not `description`)

**Add 5th column:** Add `done` to `STATUS_COLUMNS` and `TaskStatus` type. Update `normalizeStatus` to map `done`/`completed` to the `done` column (currently mapped to `review`). Completed tasks must be visible in their own column.

**Add:**
- **Create task** ‚Äî form at top: title (required), context (NOT "description" ‚Äî matches API field name), priority (low/normal/high/urgent), scope (fleet/circle). Calls `wireApiCall('POST', '/api/v1/wire/tasks', { title, context, priority, scope })`.
- **Move task between columns** ‚Äî dropdown per card with valid transitions only. Use `{ action: 'move', column: '...' }` for column moves, BUT transitions to Done use `{ action: 'complete' }` and archive uses `{ action: 'archive' }` (separate server actions with additional logic). Valid `move` transitions:
  - Backlog ‚Üí Claimed (via `{ action: 'claim' }`)
  - Claimed ‚Üí Active or Backlog
  - Active ‚Üí Review (the only valid *column move* from Active). Active can also transition to Done via `{ action: 'complete' }` ‚Äî but NOT via `{ action: 'move', column: 'done' }` (the move action's route-level validation rejects 'done' as a column value)
  - Review ‚Üí Active
  - Done ‚Üí Backlog (reopen, restricted to creator/owner)
- **"Send Back" button** ‚Äî on Claimed cards: moves to Backlog. On Review cards: moves to Active. **No Send Back on Active cards** ‚Äî the server does not allow Active‚ÜíBacklog or Active‚ÜíClaimed. Active tasks can only go forward to Review or Done.
- **Complete task** ‚Äî button on Active/Review cards. Calls `wireApiCall('PUT', '/api/v1/wire/tasks/${id}', { action: 'complete' })`.
- **Archive task** ‚Äî button on any card. Calls `wireApiCall('PUT', '/api/v1/wire/tasks/${id}', { action: 'archive' })`.
- **Task priority badges** ‚Äî color-coded (urgent=red, high=orange, normal=default, low=muted).
- **Task detail expand** ‚Äî click card to see full context, assignee, timestamps.
- **Done column management** ‚Äî "Archive All Done" button in the Done column header (loops `{ action: 'archive' }` per done task). Done column collapsed by default (click to expand). Tasks in Done >7 days get visual dimming.
- **Pillar 1 note:** tasks are operational coordination state, not contributions. Archiving a task is clearing operational state, not destroying intelligence.
- ~~**Reassign**~~ ‚Äî **DEFERRED**: no server endpoint exists for operator-level reassignment. The operator can unclaim (move to backlog) and a different agent claims. This two-step flow is documented in the UI.

**Auth note:** All task endpoints use `wireApiCall` (agent token). The operator's actions execute under the node's primary agent identity. This is correct ‚Äî the operator is directing their fleet through their agent.

**Scope note:** Circle-scoped task creation needs a `circle_id` value. For v1.1, restrict the create form to fleet scope only. Circle task creation deferred until circle management UI exists.

### 3. Fleet > Mesh ‚Äî Intent Management

**Currently:** Can declare intents and write blackboard, but can't manage them.

**Add:**
- **Release intent** ‚Äî button per intent. Calls `wireApiCall('DELETE', '/api/v1/mesh/intent?intent_id=${intentId}')`. The intent_id is a query param, NOT body. Pass `X-Wire-Thread` header from the intent's `thread_id` field (read from the intent data, omit header if no thread_id).
- **Delete blackboard key** ‚Äî button per key. Calls `wireApiCall('DELETE', '/api/v1/mesh/board?key=${encodeURIComponent(key)}')`. Query param, NOT body. X-Wire-Thread is NOT required for blackboard delete (server doesn't check it). **Pillar 1 note:** blackboard keys are ephemeral operational coordination state with TTL, not contributions. Deleting a key is clearing a shared scratchpad, not destroying intelligence.
- **Intent status indicators** ‚Äî show scope, target, age.

### 4. Network Dashboard ‚Äî Operator Overview + Review Queue

**Currently:** Pulse cards showing fleet online, tasks, circles. Read-only.

**Add:**
- **Attention alerts** ‚Äî fetch from `operatorApiCall('GET', '/api/v1/operator/overview')` (works after Phase 0). Show `attention` items: quality drops, integrity events, inactive agents, low balance, spend anomalies. Color-coded by severity (red/amber/blue). Click to navigate to relevant agent via `setMode('fleet')` then `navigateView('fleet', 'agent-detail', { pseudoId })`. If Phase 1a is not yet deployed, alert clicks should be no-ops with tooltip "Agent detail coming soon."
- **Recommendations** ‚Äî show `recommendations` from overview (redeploy idle, expand domain, etc.).
- **Review queue badge** ‚Äî show `review_queue_count` with expandable section.
- **Review queue section** ‚Äî shows held contributions from `operatorApiCall('GET', '/api/v1/operator/review-queue')`. Approve/Reject buttons per item with optional note. Calls `operatorApiCall('POST', '/api/v1/operator/review-queue/${heldId}', { action: 'approve' | 'reject', note? })`. **Note:** the `heldId` is the held-contribution row ID from the queue response, NOT the contribution_id.
- **Contribution preview** in review items ‚Äî show title, body, agent pseudonym, significance.
- **Credit monitoring** ‚Äî pool balance trend and spend rate from `overview.pool` (aggregate). Per-agent spend breakdown from `overview.agents[].economics` (NOT from pool ‚Äî pool is aggregate only). Alert when balance drops below threshold.
- Extract attention/recommendations into `src/components/dashboard/OperatorAlerts.tsx` to keep DashboardMode manageable.
- **Dual-auth merge:** DashboardMode currently fetches pulse via `wireApiCall` and corpora via `operatorApiCall`. Phase 3 adds overview via `operatorApiCall`. Pulse and overview partially overlap (both show fleet status, task counts). Strategy: overview data supplements pulse cards (adds attention alerts, recommendations, review queue, credit data), does NOT replace them. Refresh overview less frequently than pulse (overview is expensive ‚Äî N+1 queries per agent). FleetOverview similarly needs both `wireApiCall` (roster) and `operatorApiCall` (agent detail/controls) ‚Äî destructure both from context.

### 5. Activity ‚Äî Rating Controls

**Currently:** Expandable notifications with contribution detail, but no rating.

**Add:**
- **Rate contribution** ‚Äî in expanded notification view. Two 0-1 sliders: accuracy and usefulness (NOT 5-star ‚Äî server strictly validates 0-1 range). Calls `wireApiCall('POST', '/api/v1/wire/rate', { item_id, item_type: 'contribution', accuracy, usefulness })`.
- **Constraints to surface in UI:** Cannot rate own contributions. Cannot rate contributions from agents under the same operator. Must have previously accessed the contribution (the expand action already calls `wireApiCall` for contribution/[id], which satisfies the access prerequisite).
- **Flag contribution** ‚Äî dropdown: outdated_data, harmful_content, plagiarism. Calls `wireApiCall('POST', '/api/v1/wire/rate', { item_id, item_type: 'contribution', flag: '...' })`.

### 6. Search ‚Äî Contribution Actions

**Currently:** View contributions, expand to read full body.

**Add:**
- **Rate contribution** ‚Äî same 0-1 sliders as Activity (from expanded view). Same constraints.
- **Flag contribution** ‚Äî same dropdown as Activity.
- **Respond** ‚Äî button that calls `setMode('compose')` and passes the contribution ID as target. ComposeMode's target context feature (Phase 6 v1) already handles this.

### 7. Identity ‚Äî Handle Release

**Currently:** Can register handles. Cannot release them.

**BLOCKED:** No handle release endpoint exists on the Wire server. The `wire_handles` table has a `released_at` column but no API surface.

**Server-side work required:** Create `POST /api/v1/wire/handles/[id]/release` (or `DELETE`). Must verify ownership, set `released_at` and `status = 'released'`, set `cooldown_until = released_at + 90 days`.

**Add (after server endpoint exists):**
- **Release handle** ‚Äî button with **impact preview** (Pillar 23): show contribution count published under this handle, whether any pending payments reference it. Confirm. Call the release endpoint.

### 8. Compose ‚Äî View Contributions + Retraction

**Currently:** Write and publish. No drafts. Can't edit after publish.

**Add:**
- **Save as draft** ‚Äî store locally in a JSON file in the data dir (no Wire endpoint needed).
- **View my contributions** ‚Äî use `operatorApiCall('GET', '/api/v1/operator/review-feed')` which returns all contributions from all agents belonging to the operator (better than per-agent history). Show with grace window status.
- **Retract contribution** ‚Äî if within grace period. Calls `operatorApiCall('POST', '/api/v1/operator/agents/${agentUuid}/contributions/${contributionId}/retract', { reason? })`. **Note:** this endpoint expects the agent's internal UUID (not pseudo_id) as the `[id]` path param ‚Äî inconsistent with sibling routes that use pseudo_id. Get the UUID from `review-feed` response's `agent_id` field. With **impact preview** (Pillar 23): show citation count, grace period remaining. Needs Phase 0 (Bearer fallback).
