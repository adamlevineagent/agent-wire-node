## The Design Question

The app works. Every tab has content and control surfaces. But user testing revealed a structural question:

**The current tab structure is organized by *data type* (pyramids, corpora, agents, notifications). But the operator's mental model is organized by *workflow* (plan work, dispatch agents, review output, manage team).**

### The Knowledge / Understanding Split

Adam's framing: raw data is **Knowledge** (corpora, sync, documents, source material). Derived intelligence is **Understanding** (pyramids, questions, vines). These are currently mixed — corpora lives in Fleet, sync lives in Node, pyramids is its own tab.

### The Fleet Verbs

The operator's relationship with their fleet has four modes:

| Verb | What | Currently lives in |
|---|---|---|
| **Plan** | Define what needs to happen. Create tasks, set priorities, scope work. | Fleet > Tasks (create form) |
| **Dispatch** | Assign work to agents. Send agents after tasks. See who's working on what. | Fleet > Tasks (claim/move) + Fleet > Mesh (intents) |
| **Review** | Evaluate what came back. Approve/reject, rate quality, retract mistakes. | Network (review queue) + Compose (my contributions + retract) + Activity (rating) |
| **Manage** | Control the fleet itself. Pause/resume, budgets, create/archive agents. | Fleet > Fleet Overview (detail drawer + controls) |

The operator's workflow is a loop: **Plan → Dispatch → Review → Manage → Plan...**

Currently this loop is scattered across 4 tabs (Fleet, Network, Compose, Activity). The operator has to context-switch between tabs to complete one cycle.

### The Open Questions

1. **Should Fleet sub-tabs map to these four verbs?**
   - Pro: Natural workflow. One tab for the entire operator loop.
   - Con: Pulls review queue out of Network, my contributions out of Compose, rating out of Activity. Those tabs lose content.
   - Con: "Review" in Fleet might conflict with the concept of reviewing contributions in Activity (which is also about notifications, not just review).

2. **If Review consolidates into Fleet, what happens to the other tabs?**
   - Network becomes purely observational (pulse data, credit monitoring, tunnel status). No actions.
   - Compose becomes purely authoring (write contributions, request work). No viewing own work.
   - Activity becomes purely notifications + messages. No rating.
   - Is that better or worse? Simpler tabs with single responsibilities vs. fewer places to look?

3. **Should the task board separate Plan and Dispatch?**
   - Currently the kanban IS both planning and dispatching — you create tasks (plan) and move them to agents (dispatch) on the same board.
   - Could "Plan" be a task list/backlog view and "Dispatch" be the active assignment view? Or is that over-splitting?

4. **Where does Search fit?**
   - Search is the operator discovering what's on the Wire. It's not part of the fleet loop.
   - But search results feed into planning (find intelligence gaps → create tasks to fill them) and review (find agent contributions to rate/flag).
   - Should Search be integrated into the fleet workflow somehow? Or stay standalone?

5. **The "command center" question:**
   - Is Fleet actually a command center? The operator's primary workspace?
   - If so, should it be the default tab (currently Pyramids is default)?
   - Or is the operator's primary work building pyramids, with fleet management as a supporting activity?

6. **How do we handle the Knowledge/Understanding split alongside the Fleet verbs?**
   - Understanding (pyramids) and Knowledge (corpora/sync) are about the DATA.
   - Fleet (plan/dispatch/review/manage) is about the AGENTS.
   - These are orthogonal dimensions. The tab bar is one-dimensional.
   - Do we pick one organizing principle, or find a way to serve both?

### Proposed Tab Structure (under discussion)

| # | Tab | Concept |
|---|-----|---------|
| 1 | **Understanding** | Pyramids, questions, vines (the intelligence you've built) |
| 2 | **Knowledge** | Corpora + Sync merged (the raw data you're building from) |
| 3 | **Fleet** | Plan / Dispatch / Review / Manage (your agents and their work) |
| 4 | **Search** | Wire discovery (what's out there) |
| 5 | **Compose** | Authoring (what you're contributing) |
| 6 | **Network** | Infrastructure (connection, credits, tunnel, pulse) |
| 7 | **Activity** | Notifications + messages |
| 8 | **Identity** | Account (handle, reputation, transactions) |
| 9 | **Settings** | Configuration |

This groups the first 3 tabs as the operator's workspace (Understanding + Knowledge + Fleet), the next 2 as Wire interaction (Search + Compose), and the last 4 as operational/administrative.

But the Fleet verbs question is still open — do the Fleet sub-tabs become Plan/Dispatch/Review/Manage, or stay as Fleet Overview/Tasks/Coordination?

### What Needs Deciding Before Building

1. Fleet sub-tab structure (verbs vs. current)
2. Whether Review consolidates into Fleet
3. Whether the task board splits into Plan + Dispatch views
4. Node tab fate (keep Market/Remote/Logs or redistribute)
5. Default tab (Understanding or Fleet?)
