14. One-page summary

### Load-bearing design decisions (so the next agent doesn't accidentally un-make them)

These were arrived at through iteration with the human; they're canonical and should not be revisited without a strong reason:

- **The consumer is the agent, not the human.** The human has biological persistent memory. The pyramid exists to give the agent continuity (Mode A cold start) and working memory (Mode B in-session hydration loop). Every framing decision in the doc follows from this.
- **One recursive synthesis prompt, level-agnostic.** `synthesize_recursive.md` runs at L1→L2, L2→apex, and at every vine composition layer above forever. It never references layer numbers or absolute depth. It describes the operation ("one zoom level above your inputs") relative to what it sees. This means vine composition doesn't need new prompts at each scale.
- **Schema is invariant across layers and mostly optional.** Every layer has the same field shape. Only `headline`, `time_range`, `weight` are required; everything else is optional and appendable by subsequent passes. This is load-bearing for both runtime dehydration (dehydrated = node with fields dropped) AND pipeline extensibility (future passes add fields without breaking old nodes).
- **`decisions[].stance` collapses the prescriptive fields.** No separate `active_commitments` / `ruled_out` / `open_questions` / `discarded` fields. All of these are decisions in different states. One field, one sub-attribute (`stance`), N views via filter. The stance vocabulary is open (committed / ruled_out / open / done / deferred / superseded / conditional / other).
- **`importance` is a 0.0–1.0 score on topics, decisions, and quotes** that drives dehydration priority, compression priority, query ranking, and runtime eviction. One concept, multiple use cases.
- **Zoom level, not length.** Prompts prescribe the abstraction shift ("abstract one level above the inputs"), not output length. The 50% ceiling is a guardrail against degenerate expansion, not a target. Length is content-determined.
- **Quote asymmetry:** human quotes preserved as authoritative direction (the human's exact words are binding); prior-agent quotes preserved as earned state (commitments, discoveries, rulings, definitional claims); agent exposition paraphrased. In AI sessions where the AI speaks ~95% of tokens, this bias is essential.
- **Apex accuracy: the recursive prompt's purpose block phrases upward consumption as *potential* not *guaranteed*.** This makes the prompt accurate at every layer including the current top of any build. The prompt never asserts "another layer above exists" because at the apex of a single-session build, that claim would be false. The phrasing is "your output may be composed upward at any future point, and you cannot know from within the prompt whether you are at the top or in the middle."
- **Runtime integration is not an afterthought.** The pipeline exists to feed the runtime. Every earlier design decision (schema invariance, optional fields, importance, ties_to, recursive invariance, fractal structure, dehydration cascade) is load-bearing for both the build-time pipeline AND the runtime working-memory consumption.

### What Phase 2 does (recursive ask escalation)

From Section 1 of the prep-v2 doc (lines 159-620):

When a vine pyramid's `gap_processing` step finds unresolved gaps (questions that couldn't be answered from the vine's own L0 nodes), Phase 2 lets the vine **escalate the gap to the source pyramids** that the vine is composed from. Each source pyramid spawns a child question pyramid (`{source}--ask-{hash}`) that decomposes the gap question and tries to answer it from its own corpus. The results flow back into the vine's evidence loop and re-resolve the original gap.

This is recursive: a source pyramid's child may ALSO find gaps, escalate to ITS sources, and so on. Cycle prevention via a visited-set. Depth limit via `Tier3Config.vine_max_recursion_depth`. Parallel child builds via `vine_max_parallel_escalations`.

Key verified facts from Section 0 of the prep:
- `build_runner.rs::run_decomposed_build` has 10 parameters and returns `(build_id, node_count, step_activities)`. The doc specifies adding an 11th parameter `extra_initial_params: Option<HashMap<String, serde_json::Value>>` to thread recursion-depth and visited-set through (CRIT-4 from audit).
- `chain_executor.rs::execute_process_gaps` has a closure at line 5312-5351 that declares `file_slugs` and `pyramid_slugs` as local variables not visible to outer scope; the dispatcher refactor has to either return more from the closure or move the partition outside (CRIT-3).
- Child question pyramid persistence writes nodes under the CHILD slug, not the source slug — so re-resolving the parent vine's gap after escalation needs to walk the referrer graph, not look at the source slug's nodes directly (CRIT-5). The v2 fix uses a dedicated `reference_type = 'vine-escalation'` value on `pyramid_slug_references` to filter. **This is the single most load-bearing correctness fix in v2 and deserves extra attention during re-audit.**
- `Tier3Config` location confirmed; new fields `vine_max_recursion_depth`, `vine_max_parallel_escalations`, `vine_escalation_retry_on_error` added to it.

### What Phase 4-local does (cross-operator vines without payment)

From Section 2 of the prep-v2 doc (lines 621-905):

Phase 4 expands vine composition to pull in REMOTE pyramids from other operators on the Wire network. A vine can reference a remote slug at `operator-X/slug-Y`, and gap_processing can resolve evidence against the remote pyramid via HTTP calls to the remote operator's public endpoints.

**Phase 4-local** is the subset that works without payment infrastructure:

### Risks the next impl must verify (from Section "Risks" in prep-v2)

1. **CRIT-5 correctness via reference_type filter.** The referrer-graph re-resolve depends on every escalation child slug being distinguishable. The v2 recommendation is to use a dedicated `reference_type = 'vine-escalation'` value instead of a substring check on slug names. This is **strongly recommended for impl** and should be confirmed during re-audit.
2. **Question tree initialization for child builds.** The first child build will reveal whether `run_decomposed_build` requires a pre-existing `pyramid_question_trees` row. 4. **Parallel child build LLM rate limits.** `vine_max_parallel_escalations` defaults to 3. Higher values fan out faster but may hit OpenRouter rate limits.
5. **Remote query latency.** `RemotePyramidClient::remote_search` has no timeout as currently written. Add a default 30s timeout during impl.
6. **Embargoed-tier semantics for escalation.** Local evidence resolution already skips embargoed slugs, but escalation (Phase 2) currently doesn't check tiers. The prep doc recommends adding the same `tier == "embargoed"` check at the top of `escalate_gap_to_source_pyramids`.

4. **Apply findings** from both stages. If findings surface, the plan may need another iteration before implementation.
5. **Implement** the corrected plan. Cargo check at every phase boundary. Don't skip.
6. **Implementation audit** — typically one pair of informed auditors on the shipped code to catch what implementation introduced that planning didn't anticipate.
7. **Apply impl audit findings.** Cargo check again.
8. **Smoke test** against the plan's done criteria.
9. **Write a ship handoff** capturing what landed, what was deferred, which tests confirmed what, and which latent bugs stayed untouched.

The human's phrase is "audit til clean." The goal isn't a specific count of audit cycles; it's converging to a state where no auditor finds blocking issues. Most workstreams converge in 2-3 audit cycles total; some need more.

- **The human has a sharp eye for load-bearing vs decorative design decisions.** When iterating on a plan, be ready to defend every choice with "this is load-bearing because X" or drop it. The episodic memory design doc was built through 20+ iterations of this kind of challenge.
- **The human prefers being told "here's what you're pointing at" over being asked "what do you mean by X?".** When they make an insight, your first job is to crystallize it back to them with precision to confirm shared understanding. Then expand / challenge / refine.
- **"MPS?" means Maximal Potential Solution check.** Use the `mps` skill when the human asks it.
- **Same-day multi-phase ships are normal.** Don't use "ship-it-safer" or conservative-scoping language; phases land hours apart, not weeks.
- **Fix all bugs when found**, even if unrelated to the current task. This is in the global CLAUDE.md.
- **For bug fixes, run a second fixer in series instead of a full audit.** This is also in CLAUDE.md.
- **The human's exact quotes are binding.** If the human said "do X," that's direction. Paraphrased recollection is worse than dropping the paraphrase and looking it up in the transcript or handoff.

### About the architecture

- **Chain-binding-v2.6 is the current default for conversation slugs** via `default_chain_id("conversation") → "conversation-chronological"`. No per-slug assignment needed.
- **The chain executor's `pair_adjacent` primitive exists** but has been used historically with layer-specific prompts. Using it with a single recursive prompt across layers (as episodic memory requires) is new territory and may need small executor tweaks.
- **The `web` primitive runs at each depth to produce cross-cutting edges** that become the meta-navigation layer. Webbing is what makes the chronological pyramid queryable as a graph rather than just readable as a linear tree.
- **The `$item.<field>` dot navigation in `chain_resolve.rs`** is new as of this session. Forward-pass and reverse-pass steps rely on it. Any new chain step that reads chunk content via `$item.content` will work now; before this session they would have errored with "Unresolved reference: $item.content."
- **Pillar 37 is a hard rule:** prompts must NOT prescribe output sizes/lengths/counts/sentence counts/percentages. "Compress to ~50% of input" is a compression-ratio guardrail against degenerate expansion, not a target. "Output should be ~1500 chars" IS a violation.
